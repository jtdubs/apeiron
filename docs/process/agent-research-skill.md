# Agent Research Skill: Fractal Mathematics & Algorithms

This document defines the standard operating procedure (SOP) and required multi-agent workflow for researching, analyzing, and documenting complex, historically undocumented fractal rendering algorithms.

## 1. Objective

Fractal rendering algorithms (e.g., Bilinear Approximation, Series Approximation, Perturbation Theory constraints) often lack formal, centralized specification. Deep knowledge is typically scattered across obscure forums, academic PDFs, and legacy C++/Delphi reference implementations.

The goal of this skill is to deploy an asynchronous, multi-agent architecture to systematically scrape, extract, modernize, and synthesize this knowledge into formal engineering whitepapers.

## 2. Multi-Agent Pipeline Architecture

To prevent context-window exhaustion and hallucination during massive document parsing, the research process must be divided into specialized, isolated agents acting in a map-reduce style pattern.

### Phase 1: The Research Coordinator (Dispatcher)

- **Input:** Research topic (e.g., "Bilinear Approximation for Fractal Perturbation").
- **Job:** Performs broad search queries across the web, specifically targeting sources like fractalforums.org, K.I. Martin's SuperFractalThing repo, Claude Heiland-Allen's notes, etc.
- **Output:** Generates a structured list of 5-10 highly relevant target URLs (forum threads, source code files, PDFs). Does _not_ attempt to read them deeply yet.

### Phase 2: The Scribes (Data Extractors / Workers)

- **Input:** 1 Target URL from the Coordinator, 1 Research Topic.
- **Job:** This phase spins up multiple isolated agent contexts (one for each source). The Scribe visits the source, reads the content deeply, and extracts all relevant mathematical formulas, legacy hardware-specific code, and contextual clues.
- **Output:** An isolated knowledge dump (e.g., `scribe_source_1.md`). This process isolates context windows safely.

### Phase 3: The Mathematician (Analyzer)

- **Input:** The compiled outputs from all Phase 2 Scribes.
- **Job:** Synthesizes the raw data. Filters out forum noise and hardware constraints. Distills the pure mathematics, identifying core formulas. If scribes report conflicting equations from different sources, notes the discrepancy.
- **Output:** A "Math Core" artifact detailing the pure logical formulas.

### Phase 4: The Engineer (Translator)

- **Input:** The "Math Core" artifact.
- **Job:** Translates the pure mathematics into modernized, hardware-agnostic pseudocode. Strips out legacy optimizations (like SSE2 intrinsics) and focuses on algorithmic flow suitable for the Apeiron architecture (Rust / WGSL bounds).
- **Output:** A robust pseudocode summary and structural limitation analysis.

### Phase 5: The Technical Writer (Synthesizer)

- **Input:** Outputs from Phase 3 and Phase 4, alongside the required Whitepaper Schema.
- **Job:** Assembles the final Markdown document.

---

## 3. The Whitepaper Schema

The final artifact generated by Phase 5 must strictly adhere to the following structure, using standard Markdown and MathJax/KaTeX (`$$` / `$`) for formulas. It must not use LaTeX.

1.  **Abstract:** A brief, high-level summary of the algorithm.
2.  **Historical Context:** Origins, primary authors, and what the algorithm originally replaced.
3.  **The Problem:** The specific computational constraint or mathematical limitation this algorithm solves (e.g., $f64$ truncation limits).
4.  **Mathematical Mechanics:** The pure mathematical theory underlying the solution, complete with cleanly formatted Markdown math blocks.
5.  **Algorithmic Implementation:**
    - Hardware-agnostic pseudocode (modernized).
    - Expected input structures and output structures.
6.  **Domain Validity & Limitations:**
    - At what zoom depths ($10^{-x}$) or parameter ranges does this remain mathematically sound?
    - When does it mathematically break down or diverge?
7.  **Failure Modes & Diagnostics:**
    - What are the visual symptoms when the algorithm fails? (e.g., "Proxy collapse yields a solid black screen").
8.  **Application to Apeiron (Optional/Future Work):** How this maps to our explicit `[zr, zi, cr, ci]` bounds or Rust/WGSL data limits.
9.  **References/Bibliography:** Links directly to the threads, repos, and PDFs used by the Scribes.

## 4. Execution Visibility & Tracking

Because this multi-agent process is asynchronous and heavily decoupled, the agents MUST adhere to real-time visibility practices so the user is never left waiting on a silent black box.

### The Live Research Dashboard

Upon receiving a research topic, the Coordinator must immediately create a tracking artifact (e.g., `docs/tasks/active_research.md`). This document acts as the live telemetry stream for the research job.

1.  **State Table:** The dashboard must contain a Markdown table detailing every target URL and the current status of the Scribe assigned to it (e.g., `Pending`, `Extracting`, `Completed`, `Failed - 404`).
2.  **Incremental File Drops:** Scribes and Intermediate phases must not hold data in memory. They must write their intermediate files (e.g., `scribe_source_1.md`, `math_core_draft.md`) directly into the file system (typically to a scratch or task directory) as soon as they complete. This allows the user to read the extracted knowledge in real-time while subsequent phases queue up.
3.  **No Manual Gates:** Because the Research Coordinator and Mathematician are trusted to evaluate sources and equations autonomously, manual approval gates are disabled. The system will flow from Phase 1 through Phase 5 automatically, relying entirely on the Dashboard and Incremental File Drops to communicate progress.
