# Apeiron Test Plan & Debugging Methodology

## 1. Overview and Core Philosophy

Rendering fractals using perturbation theory and WebGPU introduces multiple potential failure points: arbitrary precision truncation, false interior bounds scaling, and shader compilation failures.

To solve this deterministically, Apeiron explicitly separates testing into two distinct analytical flavors, run entirely headlessly without relying on DOM elements or `requestAnimationFrame` UI logic.

## 2. The Two Flavors of Data Testing

Apeiron avoids perceptual image diffing (e.g. `pixelmatch` on PNGs) because multi-sampling (MSAA) and device-specific rendering engines create hundreds of false positives. Instead, we use **Data-First Buffer Matching** by extracting the WebGPU Compute Arrays directly into headless Node/Deno environments.

### Flavor A: Correctness Testing (Fuzzy Comparison)

- **Goal:** Prove the fast WebGPU shader math approximates the true mathematical fractal.
- **Methodology:** We compare the output array buffer generated by WebGPU against a "ground truth" buffer generated by our slow-but-perfect Rust `BigFloat` core. **Crucially, Perturbation assertions require proxy validation at scaled sub-pixel offsets (delta_c clusters), not just single analytical anchor points.** The runner synthesizes arrays of tightly scaled deviations to test fragment shader identicals.
- **Tolerance:** Because WebGPU utilizes rigid `f32` (or emulated `f64`) floats for speed, minor iteration discrepancies are mathematically inevitable at the boundaries of deep limits. Therefore, Correctness Tests evaluate bounding limits using **epsilon tolerances (Fuzzy Matching)**. As long as the WebGPU calculation does not diverge fundamentally (e.g., creating a "black hole"), it passes.

### Flavor B: Regression Testing (Bit-Perfect Comparison)

- **Goal:** Ensure refactors and architecture updates do not unexpectedly alter existing math behaviors.
- **Methodology:** We compare the WebGPU array buffer generated _today_ against a cached WebGPU array buffer generated natively on the same machine _last week_.
- **Tolerance:** WebGPU shaders are strictly deterministic. So long as the viewport inputs remain identical, the mathematical buffer output MUST be identical. Regression testing operates on **strict Bit-Perfect equality**. A single floating-point change signals a regression.

## 3. Implementation Details for Agents

### 1. Generating Ground Truth (`rust-math`)

- Ground truth test cases live in `tests/cases.json`.
- The Rust worker iterates these bounding boxes using pure `BigFloat` precision.
- Instead of outputting `.png` images, the Rust test generator dumps a JSON payload or binary ArrayBuffer containing the resolved `[Iteration, EscapedAt]` pair for each coordinate matrix.
- This creates unassailable `.json` ground truth files we commit to the repository (`tests/artifacts/truth_case_01.json`).

### 2. Headless WebGPU Execution

- The testing runner initializes `createFractalEngine()` explicitly _without_ an HTML Canvas (`document.createElement` is not needed).
- **CI / Headless Environment Adapter Checks:** Because CI environments (like GitHub Actions) natively lack physical GPUs, test runner initialization MUST include an explicit Step 0 hardware check for `navigator.gpu.requestAdapter()`. The orchestration script must explicitly pass necessary software rasterization flags (e.g., `--enable-unsafe-webgpu` for Deno or pre-linking `dawn.node`) to leverage SwiftShader/Vulkan. If an adapter fails to securely initialize, the test suite MUST print an explicit environment error and halt, preventing hundreds of downstream tests from silently failing due to missing WebGPU context.
- The engine compiles the active `WGSL` compute shaders using the acquired headless adapter.
- It runs the identical `cases.json` inputs through the GPU, maps the final GPU `StorageBuffer` back to the CPU via `buffer.mapAsync()`, and returns an extracted Float32Array.
- The test runner then applies Flavor A (Fuzzy Match vs Truth JSON) or Flavor B (Bit-Perfect Match vs old WGSL JSON).

## Recommended Debugging Workflow

1. A glitch is identified via UI usage.
2. The coordinate inputs are extracted into a new entry in `tests/cases.json`.
3. The Rust generator compiles the exact mathematical ground-truth ArrayBuffer.
4. The WGSL shader is debugged and continuously validated against Flavor A until the epsilon error bounds pass.
5. The passing WGSL array is cached to prevent future regressions via Flavor B tests.
