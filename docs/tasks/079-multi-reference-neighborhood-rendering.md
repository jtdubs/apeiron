# Task 079: Multi-Reference Neighborhood Rendering

## Objective

Transition the rendering engine from a Single-Reference shift model to a true Multi-Reference Frame architecture as detailed in K.I. Martin's Perturbation methods. This will allow the GPU to natively swap a glitched pixel to a new neighborhood branch dynamically without interrupting the rendering pass.

## Current Limitations & Rationale

Currently, Phase 1 "Chained Rebasing" works by moving the global viewport anchor to the coordinate of the detected glitch. While effective for highly dense, localized zooms, this "Single-Reference" model creates critical failures on **spatially wide chaotic boundaries**:

1. **The Tug-of-War Effect:** If the screen perfectly bisects two distinct mathematical features (e.g., one side falls deeply into the Minibrot, the other side diverges rapidly into the exterior), no single reference can adequately cover the entire screen. The GPU will glitch on the right side. The TS orchestrator shifts the anchor to the right. The next frame, the left side glitches. The orchestrator shifts left. The UI engine becomes trapped in an infinite "tug-of-war" rebase loop, never completing the frame.
2. **Costly Interruption:** Re-referencing globally forces a `-6.0` proxy collapse yield, requiring a full pipeline stall, passing the glitch buffer over `postMessage` to the Rust worker, and dropping all completed rendering progress.

**The Solution:** By moving the `ReferenceTree` evaluation directly into the WebGPU shader, each pixel can independently query the reference buffer. Pixel A resolves against Reference Branch A; Pixel B resolves against Reference Branch B. This prevents global tug-of-war stalls, entirely eliminates mid-pass TypeScript yields for known boundaries, and natively renders chaotic "multi-neighborhood" screens in a single performant pipeline execution.

## Requirements

- **GPU Reference Tree Traversal:** The WebGPU shader (`core_compute.wgsl`) must accept a packed buffer representing the `ReferenceTree` generated by Rust, containing multiple reference orbits and their valid coordinate bounds.
- **Dynamic Branch Re-referencing:** When a WebGPU thread detects a proxy collapse (`p_mag < dz_mag`), instead of yielding `-6.0` and aborting to TypeScript, it must search the `ReferenceTree` buffer for a child reference orbit that spatially covers the glitched pixel.
- **Local Geometric Offset Translation:** Once a new reference is selected mid-pass, the shader must safely recalculate the local $\delta z_n$ taking into account the vector offset between the parent orbit and the child orbit geometry, allowing iteration to seamlessly continue without precision loss.
- **Fallback Glitch Report:** Only jump out to the `-6.0` return code (reporting to TypeScript) if the pixel proxy-collapses and mathematically falls entirely outside all pre-calculated `ReferenceTree` boundary zones.

## Implementation Plan

1. In `rust-math/src/reference_tree.rs`, write logic to serialize the in-memory tree nodes and their BLA boundaries into a structured flat `Float64Array` layout.
2. In `MemoryLayout.json`, add a schema for `ReferenceTreeArray` (or update `OrbitMetadata`) to handle multiple reference orbits concurrently.
3. Update `rust.worker.ts` to surface the structured `ReferenceTree` array to the orchestration layer.
4. Update `WebGPUTestHarness` and `PassManager` to bind the tree buffer.
5. In `core_compute.wgsl`, modify the proxy-collapse detection block in `perturbation_step`. Implement the tree traversal search logic, and `break` into the new reference frame if a match is found.
6. Ensure headless tests explicitly map scenarios where multiple neighborhood references are correctly matched and executed natively.
